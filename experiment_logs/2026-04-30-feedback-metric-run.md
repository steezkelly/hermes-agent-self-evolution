# Experiment: Rich Feedback Metric — 2026-04-30

## Hypothesis
Returning diagnostic feedback text alongside the score (instead of a bare float) gives GEPA's reflection LM useful signal about WHY a section's instruction text led to poor output, enabling better mutations.

## Change
Modified `skill_fitness_metric` in `evolution/core/fitness.py`:
- **Before**: `-> float` — GEPA's `feedback_fn_creator` wrapped this into `{"score": o, "feedback": "This trajectory got a score of X."}` (useless for reflection)
- **After**: When `pred_name is not None` (GEPA requesting per-predictor feedback), returns `{"score": float, "feedback": str}` where feedback is generated by `_generate_feedback()` — keyword overlap analysis with zero extra LLM cost.

### `_generate_feedback` logic
1. Empty output → "predictor produced empty output for task: ..."
2. Missing > 3 key terms with < 30% overlap → "missed most of expected content — discussed different topics"
3. Missing > present → "covered some ground but omitted key terms like: ..."
4. Missing exists → "good direction but missed specific details like: ..."
5. No missing → "response matches expected content well"
6. Skill body invoked (raw > 500 chars with ## headers) and output lacks Hermes-specific terms → "lacked concrete Hermes CLI commands or references"

## Run Config
- `--iterations 12 --optimizer-model deepseek-v4-pro --eval-model deepseek-v4-pro`
- Skill: `hermes-agent` (29,929 chars, 11 sections)
- `component_selector` default (round-robin)
- Killed ~65 min in at iteration 15/12 (budget exhausted), not all v2.1 pipeline steps completed

## Results

| Metric | Before (no feedback) | With feedback |
|--------|---------------------|---------------|
| Baseline valset score | 0.620 | 0.571 |
| Accepted mutations | 1/6 (17%) | **3/6 (50%)** |
| Best valset score | 0.634 | **0.634** |
| Pareto front size | 2 candidates | **4 candidates** |
| Subsample delta (best iter) | +0.028 (iteration 1) | **+0.062** (iteration 6) |

### Accepted mutations
1. **Iteration 1**: `pred_preamble` — accepted. Baseline 0.571 → program 1: 0.584
2. **Iteration 4**: `pred_slash-commands-in-session` — accepted. Program 1: 0.584 → program 2: 0.603
3. **Iteration 6**: `pred_security-privacy-toggles` (likely, round-robin order) — accepted. Program 1: 0.584 → program 3: **0.634**

### Rejected mutations
- Iterations 2: quick-start, 3: cli-reference, 5: key-paths, 8-14: cycling over best candidates — all subsample scores below baseline.

## Key Insights
1. **Rich feedback works**: Acceptance rate tripled vs the "This trajectory got a score of X" default. The reflection LM can now target specific deficiencies.
2. **Round-robin is the right selector**: All 3 accepted mutations were focused, single-section changes. The "all" selector run (earlier) produced 0 acceptances.
3. **GEPA's Pareto selector correctly re-visited best candidates**: Iterations 4 and 6 selected program 1 (not program 0) for re-mutation after it had already been improved.
4. **Budget exhaustion**: 12 iterations ≈ 120 metric calls with round-robin = ~11 single-section mutation attempts + eval. With the slow reflection LM (deepseek-v4-pro, ~3-5 min per proposal call), the process takes ~65+ minutes.

## Open Issues
- v2.1 holdout checker: best valset score 0.634 doesn't guarantee holdout improvement (needs 10+ holdout examples for statistical significance)
- Evolved body extraction: `multi_component_extract` relies on `evolved_sections()` which reads back `pred.signature.instructions`. If GEPA's `build_program()` mutates signatures correctly, this works — but verified evolved body was same 29,929 chars in some runs, suggesting a readback issue.
- Ollama Cloud backend flakiness: some runs return flat 0.3 scores across all examples.
